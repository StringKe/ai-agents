# 测试反模式

## 1. 冰淇淋锥反模式

大量端到端测试、少量单元测试。

**问题**：执行慢、失败难定位、维护成本高。

**正确做法**：测试金字塔。大量单元测试、适量集成测试、少量端到端测试。

**检查**：如果运行全部测试需要超过 2 分钟，大概率存在这个问题。

## 2. Mock 过度

Mock 了太多东西，测试只验证了 mock 的行为，不验证真实行为。

**问题**：测试通过但生产环境失败。mock 和真实实现不一致时，测试无法发现。

**正确做法**：
- 只 mock 外部依赖（第三方 API、数据库、文件系统）
- 自己的代码用真实实现
- 如果需要 mock 太多东西才能测试，说明设计有问题

**检查**：一个测试文件中 mock 数量超过 3 个时，审视是否过度 mock。

## 3. 测试实现细节

测试耦合了内部实现（私有方法、内部数据结构、调用顺序）。

**问题**：重构时测试大量失败，但行为实际没变。导致团队害怕重构。

**正确做法**：
- 通过公共接口测试
- 验证输出和副作用，不验证内部过程
- 问自己："如果我重构了内部实现但保持行为不变，这个测试还能通过吗？"

**检查**：如果重命名一个私有方法会导致测试失败，说明测试了实现细节。

## 4. 测试中的条件逻辑

测试代码中出现 if/else、循环、try/catch。

**问题**：测试本身可能有 bug。测试应该是直线执行的简单断言。

**正确做法**：
- 每个测试只有一条执行路径
- 用参数化测试替代循环
- 用独立的测试用例替代 if/else

**检查**：测试文件中搜索 `if`、`for`、`while`，如果不是在 helper 函数中，就是反模式。

## 5. 共享可变状态

多个测试共享同一个可变对象，执行顺序影响结果。

**问题**：测试单独运行通过，一起运行失败（或反过来）。

**正确做法**：
- 每个测试自己创建所需的数据
- 用 beforeEach/setUp 重置状态
- 不依赖测试执行顺序

**检查**：随机打乱测试执行顺序，如果有测试失败，说明存在共享状态问题。
