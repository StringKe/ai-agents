# /receiving-code-review — 处理代码审查反馈

## 核心原则

**技术正确性高于社交舒适。**

- 不做客套回复（"好的建议！"、"谢谢指出！"）
- 每条反馈独立评估，基于技术事实
- 该接受的立即修复，该拒绝的给出技术理由

## 处理流程

### 第一步：分类反馈

将每条审查反馈分为：

1. **有效**：指出了真实的技术问题，需要修复
2. **过度设计**：建议引入不必要的复杂度（YAGNI 检查）
3. **主观偏好**：代码风格差异，非技术问题
4. **误解**：审查员对上下文的理解有误

### 第二步：处理有效反馈

按严重度排序处理：
1. 关键问题：立即修复
2. 重要问题：修复或给出不修复的技术理由
3. 建议：评估后决定是否采纳

每个修复都要验证：编译通过、测试通过。

### 第三步：回应拒绝

对于不接受的反馈，提供：
- 具体的技术理由（不是"我觉得没必要"）
- 相关的代码上下文
- 如果适用，引用项目的设计决策或架构约定

### 第四步：汇总回复

向用户报告处理结果：

```
## 审查反馈处理

### 已修复
- [反馈内容] → [修复方式]

### 已拒绝
- [反馈内容] → [拒绝理由]

### 需要讨论
- [反馈内容] → [需要讨论的原因]
```

## YAGNI 检查

对每条"建议添加"类型的反馈，问：

1. 当前有具体的使用场景吗？
2. 不做这个改动，现有功能是否受影响？
3. 添加后的维护成本是否超过收益？

如果答案是"没有 / 不受 / 是"，则拒绝并说明理由。

## 回退协议

如果审查发现的问题指向更深层的架构问题，不要在当前 PR 中修补：
1. 记录问题
2. 建议在独立分支中处理
3. 在当前 PR 中只修复表层问题
