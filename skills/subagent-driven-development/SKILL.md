# /subagent-driven-development — 子代理驱动开发

为 plan.md 中的每个任务派遣一个独立的子代理执行。每个子代理拥有完整的任务上下文，执行完成后经过两阶段审查确保质量。

## 前置条件

- 存在已批准的 plan.md
- plan.md 中包含 Todo List

## 核心机制

### 子代理隔离

每个子代理：
- 只接收当前任务的上下文（plan.md 全文 + 相关文件路径）
- 独立完成实现、测试、编译验证
- 通过状态码报告执行结果

### 两阶段审查

任务完成后，依次派遣两个审查子代理：

1. **规格合规审查**：检查实现是否与 plan.md 的任务描述一致
2. **代码质量审查**：检查代码质量、架构合规、安全性

两阶段都通过后，任务才标记为完成。

## 流程

### 第一步：加载计划

读取 plan.md，提取所有待完成任务（`- [ ]`）。

### 第二步：逐任务派遣

对每个任务：

1. 使用 Agent 工具派遣实现子代理，prompt 结构：
   - 引用 `references/implementer-prompt.md` 的内容
   - 附加 plan.md 全文
   - 指定当前任务编号和描述
   - 列出相关文件路径

2. 解析子代理返回状态：
   - **DONE**：进入审查阶段
   - **DONE_WITH_CONCERNS**：进入审查阶段，审查时重点关注子代理提出的疑虑
   - **NEEDS_CONTEXT**：提供额外上下文后重新派遣
   - **BLOCKED**：向用户报告阻塞原因，等待指示

### 第三步：两阶段审查

对每个完成的任务：

1. 派遣规格审查子代理（使用 `references/spec-reviewer-prompt.md`）
   - 检查实现是否与任务描述一致
   - 检查是否有遗漏或超出范围的改动

2. 派遣代码质量审查子代理（使用 `references/code-quality-reviewer-prompt.md`）
   - 检查代码质量、命名、错误处理
   - 检查架构合规性和安全性

3. 如果审查发现问题：
   - 关键问题：派遣修复子代理
   - 重要问题：记录并向用户报告
   - 建议：记录但不阻塞

### 第四步：更新进度

每个任务审查通过后，在 plan.md 中标记为 `[x]`。

### 第五步：收尾

所有任务完成后：
1. 运行完整编译和测试
2. 向用户报告完成情况，包含各任务审查结果摘要

## 模型选择

- 实现子代理：使用默认模型
- 审查子代理：使用默认模型
- 修复子代理：使用默认模型

## 与直接执行的关系

子代理驱动适用于：
- 任务数量多（10+）
- 任务之间相对独立
- 需要严格的质量把关

小型、强依赖的改动使用 /executing-plans 更高效。
