# ai-agents 插件概览

## 解决什么问题

AI 辅助开发的主流用法是"把需求扔给 AI，让它写代码"。这在小任务上有效，但在涉及多文件、多模块、需要设计决策的任务上，AI 的输出质量急剧下降。原因不是模型能力不够，是上下文管理失控：AI 在编码过程中同时做架构选择、API 设计、错误处理等决策，这些决策分散在代码提交中，没有集中记录，后续维护和 Code Review 时信息丢失。

SDLC（Software Development Life Cycle，软件开发生命周期）工作流的核心思路：把决策和执行拆开。先用 AI 做深度调研和方案设计（产出 spec.md），再把方案转化为可执行的细粒度计划（产出 plan.md），最后让 AI 按计划逐步实现。决策发生在计划阶段，实现阶段只是机械执行。

收益：犯错更少、决策更早、反馈更快。

## 为什么选 Claude Code

本插件依赖 Claude Code 的几个独有能力：

**四级上下文分层**。全局（`~/.claude/CLAUDE.md`）→ 项目（`CLAUDE.md`）→ 规则（`rules/*.md`）→ 模块（`src/.../CLAUDE.md`）。其他 AI 编码工具（Codex CLI、Gemini CLI、Kimi Code）把所有规则塞在一个文件里，规则增长后变得难以维护且浪费 token。本插件用 Rule 做感知层（精简，每次加载），Skill 做执行层（详细，按需加载），这种分离只有在 Claude Code 的分层体系下才自然。

**Skills 框架**。可复用的自动化流程，支持 YAML frontmatter 声明触发条件和工具权限。AI 根据 description 自动匹配用户意图，或通过 `/name` 手动触发。本插件的 16 个技能全部构建在这个框架上。

**Agent 子代理**。派遣独立子代理执行隔离任务，每个子代理拥有独立上下文。本插件用它实现任务级隔离执行、两阶段代码审查（规格合规 + 代码质量）、多任务并行调度。

**Memory**。跨会话持久记忆。AI 记住你的角色、技术背景、偏好和历史反馈，不需要每次会话重复说明。

## 设计理念

### Rule 做感知层，Skill 做执行层

**Rule**（`rules/sdlc-workflow.md`）在每次会话启动时自动加载。它的职责是识别用户意图，引导 AI 选择合适的 Skill。Rule 不包含详细的执行指令，只包含阶段到技能的映射表和核心约束。token 开销控制在最小限度。

**Skill** 按需加载，包含详细的执行流程。每个 Skill 聚焦一个具体职责，有明确的输入、输出和流程步骤。

新增能力只需添加 Skill，不需要修改 Rule。

### 铁律驱动

关键技能定义了不可违反的铁律：
- 没有失败测试就没有生产代码（TDD）
- 没有根因就没有修复（调试）
- 没有验证证据就不宣称完成（验证）
- 审批前不写代码（SDLC）

铁律比指南更有效。AI 在遵循指南时会权衡利弊做出妥协，在遵循铁律时不会。

### 上下文工程

AI 的输出质量 = f(上下文质量)。1M token 窗口缓解了容量焦虑，但上下文工程的核心逻辑没变：用最少的 token 传递最关键的信息。

成本体现在四个维度：
1. **金钱**：按 token 计费，上下文越大，每次请求费用越高
2. **注意力稀释**：上下文越长，模型对关键信息的注意力权重越低
3. **响应速度**：输入 token 数量直接影响响应延迟
4. **噪声积累**：窗口更大意味着无效 token 积累的空间也更大

本插件的沟通风格约束（禁止表情符号、禁止道歉、禁止关联词）就是上下文工程的实践：一个 emoji 消耗 2 到 3 个 token 但不传递精确信息，"抱歉，我之前理解有误"每次浪费 20 到 30 个 token 且零信息增量。去掉这些装饰性 token，AI 被迫用文字精确表达。

## 技能体系

16 个技能分为四类：

| 类别 | 技能数 | 覆盖范围 |
|------|--------|---------|
| SDLC 核心 | 4 | 需求探索、计划编写、执行实现、子代理开发 |
| 工程实践 | 3 | 测试驱动开发、系统化调试、完成前验证 |
| 协作 | 5 | 代码审查（请求/接收）、Git Worktree、分支收尾、并行调度 |
| 工具 | 4 | 技能编写、会话整理、日报、周报 |
